PaperNormAI 知识治理规范

▎ 本文档定义 PaperNormAI 知识库的维护规则、刷新触发条件、源优先级、漂移处理和更新机制。所有知识库相关操作均应遵守本文档约束。

1. 文档定位

1.1 本文档解决什么问题

知识库如果没有治理机制，会快速失去可信度。本文档要解决的核心问题是：

1. 什么情况下必须刷新知识库？
2. 当代码、蓝图、知识库三者冲突时，以谁为准？
3. 如何标记知识的新鲜度和可信度？
4. 如何处理知识漂移？
5. 谁负责维护哪些知识文档？

1.2 本文档与其他文档的关系

- .github/copilot-instructions.md：定义 AI 行为约束
- 本文档：定义知识库维护约束
- 000-doc-map.md：定义知识导航入口
- 其他知识文档：承载具体工程认知

2. 知识源优先级

2.1 三层知识源

PaperNormAI 的工程认知来自三个层次：

1. 代码（Code）：当前事实源
2. 蓝图（Blueprint）：目标约束源
3. 知识库（Knowledge Base）：可维护认知层

2.2 冲突时的优先级规则

当三者冲突时，按以下规则判断：

┌──────────────────┬──────────┬──────────────────────────────────────────┐
│       场景       │  优先级  │                 处理方式                 │
├──────────────────┼──────────┼──────────────────────────────────────────┤
│ 代码与蓝图冲突   │ 蓝图优先 │ 代码偏离了设计意图，应修正代码或更新蓝图 │
├──────────────────┼──────────┼──────────────────────────────────────────┤
│ 代码与知识库冲突 │ 代码优先 │ 知识库过时，应更新知识库                 │
├──────────────────┼──────────┼──────────────────────────────────────────┤
│ 蓝图与知识库冲突 │ 蓝图优先 │ 知识库未同步蓝图变更，应更新知识库       │
├──────────────────┼──────────┼──────────────────────────────────────────┤
│ 三者一致         │ 无冲突   │ 正常状态                                 │
└──────────────────┴──────────┴──────────────────────────────────────────┘

2.3 特殊情况

如果发现：

- 代码实现了蓝图明确禁止的内容 → 标记为架构违规，优先修正代码
- 蓝图本身存在设计缺陷 → 提出蓝图修订建议，经评审后更新蓝图
- 知识库记录了已废弃的设计 → 在知识库中标记为"已废弃"，不要直接删除

3. 知识刷新触发条件

3.1 必须刷新知识库的情况

以下变更必须评估是否需要更新知识库：

1. 新增或重构核心模块
  - 影响文档：100-system-overview.md、000-doc-map.md
  - 可能影响：对应专题文档
2. API 契约变化
  - 影响文档：400-api-architecture.md
  - 可能影响：800-cross-layer-call-chains.md
3. 数据库模型变化
  - 影响文档：200-database-models.md
  - 可能影响：600-domain-models.md
4. 领域模型变化
  - 影响文档：600-domain-models.md
  - 可能影响：300-backend-kernel-services.md
5. 规则引擎行为变化
  - 影响文档：300-backend-kernel-services.md
  - 可能影响：700-capability-map.md
6. 模板系统结构变化
  - 影响文档：300-backend-kernel-services.md
  - 可能影响：700-capability-map.md
7. 跨层调用链变化
  - 影响文档：800-cross-layer-call-chains.md
8. Agent / skill 行为约束变化
  - 影响文档：.github/copilot-instructions.md
  - 可能影响：本文档

3.2 不需要刷新知识库的情况

- 单纯的 bug 修复（不改变模块职责）
- UI 样式调整
- 文案修改
- 测试代码补充
- 注释完善
- 性能优化（不改变接口契约）

{当前已确认的架构事实}

## 4. {主体内容}

{关键模块 / 关键对象 / 关键流程}

## 5. 与其他文档的关联

- 前置文档：{必须先读哪些文档}
- 相关文档：{可能需要参考哪些文档}

## 6. 当前已知边界

{哪些部分已经稳定，哪些部分还在演进}

## 7. 待确认问题

{哪些内容还需要进一步验证}

## 8. 更新记录

4.2 为什么要这样设计

这样设计的目的是：

- AI 在不同文档间切换时有稳定的阅读预期
- 明确标记知识的新鲜度和可信度
- 记录知识的来源和依据
- 区分"已确认事实"和"待确认假设"

5. 知识新鲜度管理

5.1 新鲜度等级定义

┌────────┬────────────────────────────────────┬────────────────────┐
│  等级  │                定义                │      标记方式      │
├────────┼────────────────────────────────────┼────────────────────┤
│ 高     │ 最近 7 天内复核过，与代码一致      │ 当前可信度：高     │
├────────┼────────────────────────────────────┼────────────────────┤
│ 中     │ 最近 30 天内复核过，可能有局部过时 │ 当前可信度：中     │
├────────┼────────────────────────────────────┼────────────────────┤
│ 低     │ 超过 30 天未复核，或已发现冲突     │ 当前可信度：低     │
├────────┼────────────────────────────────────┼────────────────────┤
│ 已废弃 │ 对应设计已不再使用                 │ 当前可信度：已废弃 │
└────────┴────────────────────────────────────┴────────────────────┘

5.2 新鲜度衰减规则

- 每次代码变更后，相关知识文档的可信度自动降级一档
- 每次复核后，可信度恢复为"高"
- 如果发现知识与代码冲突，立即标记为"低"

5.3 复核依据记录

每次复核必须记录：

**最近复核时间**：2026-04-28

**复核依据**：
- 代码范围：`backend/app/domain/validation/`
- 提交：`abc1234`
- 参考文档：`docs/architecture/2026-04-28-mvp-engineering-blueprint.md`

**当前可信度**：高

**待确认点**：无

6. 知识漂移处理

6.1 什么是知识漂移

知识漂移指：知识库记录的内容与实际代码或蓝图不一致的状态。

6.2 漂移的三种类型

1. 代码先行漂移：代码已变更，知识库未更新
2. 蓝图先行漂移：蓝图已更新，代码和知识库未跟进
3. 知识孤岛漂移：知识库记录了不存在的设计

6.3 漂移处理流程

类型 1：代码先行漂移

1. 识别受影响的知识文档
2. 阅读变更后的代码
3. 更新知识文档的"核心事实"部分
4. 更新"最近复核时间"和"复核依据"
5. 在 900-learning-log.md 中记录本次更新

类型 2：蓝图先行漂移

1. 在知识文档的"待确认问题"中标记
2. 说明"蓝图已更新为 X，但代码尚未实现"
3. 可信度标记为"中"
4. 等待代码实现后再更新为"高"

类型 3：知识孤岛漂移

1. 不要直接删除
2. 在文档顶部标记：> ⚠️ 本设计已废弃，当前实现见 {新文档}
3. 可信度标记为"已废弃"
4. 保留文档作为历史参考

8. 知识库不是什么

8.1 知识库不是代码镜像

知识库不应该：

- 逐文件复制代码内容
- 变成注释型代码备份
- 记录完全可以通过 grep 得到的低价值细节

8.2 知识库不是静态文档

知识库不应该：

- 写完就不再维护
- 只增不减
- 把过时内容留在主体部分

8.3 知识库不是需求文档

知识库不应该：

- 记录"应该做什么"（那是蓝图的职责）
- 记录"为什么要做"（那是 PRD 的职责）
- 记录"怎么做"（那是实现计划的职责）

8.4 知识库应该是什么

知识库应该记录：

- 架构上的事实
- 分层关系
- 调用链理解
- 模块职责
- 功能与代码的映射
- 约束与例外
- 知识缺口与演进轨迹

9. 知识库的成功标准

9.1 短期成功标准（1 个月内）

- 所有核心模块都有对应的知识文档
- 000-doc-map.md 能回答"我要做 X，应该先读什么"
- 每次非 trivial 开发前，AI 能找到相关知识入口
- 知识库的可信度标记真实反映新鲜度

9.2 中期成功标准（3 个月内）

- 同一模块的认知不会在多次任务中重复从零建立
- 代码实现能更贴近既有架构与分层规则
- 知识漂移能在 1 周内被识别和修正
- 学习日志和技能运行日志形成可回放的决策轨迹

9.3 长期成功标准（6 个月内）

- 新加入的 AI 工具或协作成员能通过知识库快速上手
- 架构决策、问题判断、修复经验都能被保存下来
- 知识库成为团队公共认知的一部分，不只是 AI 备忘录
- 知识库的维护成本低于它带来的价值

本文档的每次重大修改应：

- 在文档顶部记录修改日期和原因
- 在 git commit message 中明确标注
- 在团队内部同步变更内容

---
11. 总结：核心原则

1. 代码是事实，蓝图是方向，知识库是桥梁
2. 知识必须标记新鲜度，不能假装永远正确
3. 漂移不可怕，不处理漂移才可怕
4. 知识库不是代码镜像，而是架构认知
5. 维护知识库是所有 AI 参与者的共同职责

当你不确定某份知识是否可信时，默认选择：先复核，再使用。
