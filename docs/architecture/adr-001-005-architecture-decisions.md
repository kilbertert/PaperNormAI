# PaperNormAI 架构决策记录

## ADR-001: 使用 docling 作为文档解析引擎

**日期：** 2026-05-03
**状态：** 已接受

### 背景

PaperNormAI 需要将用户上传的 `.docx` 文档（规范手册和论文）解析为结构化数据，供后续 AI 处理。

### 决策

使用 [docling](https://github.com/docling-project/docling.git) 作为 Python 库集成到后端，进行文档解析。

### 理由

1. **成熟开源项目** — docling 是专门用于文档结构化解析的开源项目，社区活跃
2. **结构化输出** — 能够输出统一的结构化表示，方便后续 AI 处理
3. **Python 库集成** — 可直接作为 Python 库集成到后端，不需要额外的 REST 服务
4. **统一处理** — 统一处理 spec_doc 和 thesis_doc 两种文档

### 后果

- 需要在项目中引入 docling 依赖
- 需要学习 docling 的 API 接口和使用方式
- docling 的解析能力决定了我们能从文档中提取的信息边界

## ADR-002: 使用 AI 语义理解提取规则，而非结构化规则

**日期：** 2026-05-03
**状态：** 已接受

### 背景

规范手册（spec_doc）包含格式规范，用户希望 AI 从中提取规则来校验论文。但规范文档的格式描述是自然语言，无法可靠地转化为结构化规则（如 `{"margin": {"top": "30mm"}}`）。

### 决策

AI 通过语义理解从规范手册中提取**抽象描述性规则**，而非结构化参数规则。

### 理由

1. **规范文档的模糊性** — 规范文档中很多描述无法用结构化参数覆盖（如"标题尽量不采用英文缩写词"）
2. **语义理解的优势** — AI 可以理解自然语言描述的规范含义
3. **避免过度设计** — 结构化规则可能导致信息丢失或覆盖不全
4. **与 AI-Word-Skill 配合** — 两者都依赖 AI 的语义能力，保持一致性

### 后果

- 规则存储为文本/语义描述，而非结构化参数
- 校验过程是 AI 语义比对，而非规则引擎比对
- 需要设计合适的 prompt 来引导 AI 提取和校验

## ADR-003: 使用 AI-Word-Skill 实现文档修正合并

**日期：** 2026-05-03
**状态：** 已接受

### 背景

用户在校验报告出来后，需要将修正内容合并回原始 `.docx` 文档。关键要求是"只改文字，保留排版"。

### 决策

使用 [AI-Word-Skill](https://github.com/sgsss998/AI-Word-Skill.git) 项目实现文档修正合并。

### 理由

1. **需求完美契合** — AI-Word-Skill 的核心能力就是"只改文字，保留排版"
2. **成熟项目** — 已有具体实现，不需要自己开发
3. **保留样式** — 原有的字体、段落、页面布局、样式模板都能保留
4. **公式/表格/插图** — Phase 1 不处理这些元素，但该项目为 Phase 2 提供了扩展基础

### 后果

- 需要集成 AI-Word-Skill 到项目中
- Phase 1 只处理基础文本格式的修正
- 公式、表格、插图的合并留待 Phase 2

## ADR-004: MVP 分两个阶段实施

**日期：** 2026-05-03
**状态：** 已接受

### 背景

完整的论文格式校验和修正涉及多种内容类型：文本格式（字体、字号、段落等）、公式、表格、插图。在有限资源下需要分阶段实施。

### 决策

MVP 分两个阶段：

**Phase 1（当前实施）：**
- 字体、字号、行距、段前段后、标题层级、页边距等基础文本格式

**Phase 2（后续实施）：**
- 公式格式检测与修正
- 表格格式检测与修正
- 插图格式检测与修正
- 参考文献格式检测
- 引用一致性检测

### 理由

1. **聚焦核心** — Phase 1 覆盖最常用的基础格式问题
2. **降低复杂度** — Phase 1 不涉及公式、表格、插图等复杂元素的处理
3. **快速验证** — Phase 1 完成后可以快速验证核心流程
4. **资源合理配置** — Phase 2 的复杂度更高，需要更多时间

### 后果

- 架构设计需要考虑 Phase 1/2 的扩展性
- Phase 1 的成果可独立交付和使用
- Phase 2 的实现依赖 Phase 1 的验证反馈

## ADR-005: 规则存储采用用户级别持久化

**日期：** 2026-05-03
**状态：** 已接受

### 背景

用户上传规范手册后，AI 提取的规则需要用于校验同一用户的多次论文上传。

### 决策

用户上传规范手册后，提取的规则随用户 Session 存在，同时持久化存储以备后续复用。

### 理由

1. **用户体验** — 用户不需要每次上传论文都重新上传规范手册
2. **资源优化** — 避免重复调用 AI 提取相同规则
3. **规则复用** — 同一用户的多次论文可以使用相同的规则集

### 后果

- 需要设计规则存储的数据模型
- 需要考虑规则的生命周期管理
- 规则按用户隔离，保证隐私
