# PaperNormAI AI 工程协作体系蓝图

> 本文档定义 PaperNormAI 在进入正式功能开发前的 AI 工程协作体系，用于统一规则治理、知识学习、知识沉淀、开发执行、技能编排、文档映射与日志审计。后续所有 AI 参与的工程活动，都应以本文档为流程蓝图。

## 1. 文档定位

### 1.1 这份文档解决什么问题

PaperNormAI 的目标不是“让 AI 直接写代码”，而是先把工程边界、架构认知、业务理解、执行约束和经验沉淀外部化，形成一套 AI 可读、可学、可复用、可审计的工程系统。

这份蓝图解决的是以下问题：

1. 如何让 AI 在开发前先理解工程，而不是一上来就改代码。
2. 如何把架构设计、模块划分、接口定义、业务流程等知识系统化沉淀。
3. 如何让不同任务通过 agent 与 skill 固化为稳定动作，而不是每次临场发挥。
4. 如何让每次学习、评估、开发、修复都留下日志和回放路径。
5. 如何降低“知识只存在脑子里”带来的协作风险、重复沟通成本和架构漂移风险。

### 1.2 本文档与其他文档的关系

当前已有文档中：

- `docs/architecture/2026-04-28-mvp-engineering-blueprint.md`：定义 PaperNormAI 的产品与工程架构边界。
- `docs/superpowers/plans/2026-04-28-paper-norm-ai.md`：实现计划草案。

本文档的定位是：

- 不替代 MVP 工程架构蓝图。
- 不直接承担功能实现计划职责。
- 专门定义 **AI 如何理解工程、如何进入开发、如何沉淀知识、如何记录动作**。

当出现冲突时：

1. **产品范围、架构边界** 以 `2026-04-28-mvp-engineering-blueprint.md` 为准。
2. **AI 协作流程、知识沉淀路径、日志机制** 以本文档为准。

## 2. 体系设计目标

## 2.1 总目标

构建一套围绕 PaperNormAI 的 AI 工程协作体系，让 AI 不只是代码生成器，而是：

- 工程规则的遵循者
- 架构知识的学习者
- 公共认知的沉淀者
- 功能开发的执行者
- 缺陷修复的复盘者
- 跨文档、跨模块、跨流程的导航器

## 2.2 具体目标

这套体系至少要实现以下 8 个目标：

1. **先认知、后开发**：非 trivial 任务在开发前必须先经过知识学习或知识确认。
2. **外部化工程经验**：把原本只能由人记忆的经验变成结构化文档与规则。
3. **复用架构理解**：相同模块、相同调用链、相同业务上下文不应反复重新解释。
4. **减少架构漂移**：AI 写出的实现要尽量贴合既有工程边界，而不是临时创造新结构。
5. **动作标准化**：问题评估、就绪性检查、功能开发、缺陷修复都应有稳定技能模块。
6. **日志可回放**：每次学习、评估、开发、修复都要有记录、有入口、有上下文。
7. **知识可导航**：任意模块、功能、流程都能通过映射文档快速找到对应知识入口。
8. **知识可刷新**：知识库不是静态文档，必须具备随代码演进更新的机制。

## 3. 总体架构：四层闭环系统

在你的六步设想上，推荐抽象为四层协作系统：

1. **治理层（Governance Layer）**
2. **认知层（Cognition Layer）**
3. **执行层（Execution Layer）**
4. **导航审计层（Navigation & Audit Layer）**

它们共同组成一个闭环：

```text
治理层定义边界
  -> 认知层学习工程
  -> 知识库沉淀公共认知
  -> 执行层基于知识进入开发/修复
  -> 导航审计层记录、映射、回放、校正
  -> 反向推动治理层和认知层更新
```

### 3.1 四层与六步的映射关系

| 你的六步 | 对应层 | 核心产物 |
|---|---|---|
| 第一步：copilot-instructions | 治理层 | `.github/copilot-instructions.md` |
| 第二步：知识学习智能体 | 认知层 | `PaperNormAI-knowledge-builder.agent.md` |
| 第三步：知识库沉淀 | 认知层 + 导航审计层 | `PaperNormAI-knowledge/` 文档体系 |
| 第四步：功能开发智能体 | 执行层 | `PaperNormAI-feature-development.agent.md` |
| 第五步：技能模块化 | 执行层 | issue/readiness/feature/fix skills |
| 第六步：文档映射管理 | 导航审计层 | `000-doc-map.md` + 各类日志 |

### 3.2 体系核心原则

1. **先定义规则，再允许开发。**
2. **先建立认知，再产生实现。**
3. **先沉淀公共知识，再复用知识指导动作。**
4. **每个动作都应留下可审计痕迹。**
5. **任何知识如果不能被定位、追踪、刷新，就不能视为稳定资产。**

## 5. 治理层蓝图

治理层的作用是先回答：

- AI 能做什么？
- AI 不能做什么？
- 哪些边界不能越？
- 哪些变更必须先确认？
- 哪些动作必须留下日志？
- 哪些情况必须回写知识库？

治理层的核心载体是：

- `.github/copilot-instructions.md`

## 5.1 `copilot-instructions.md` 的职责

它不应只是泛泛而谈的“代码规范文件”，而应该是 AI 的工程治理入口。

建议至少覆盖以下 10 类内容：

1. **产品边界**：当前 MVP 做什么、不做什么。
2. **架构边界**：模块化单体、规则优先、AI 为辅、中间文档模型优先。
3. **分层规则**：哪些模块可以依赖哪些模块，哪些依赖不允许出现。
4. **目录规则**：新增文件应放在哪里，不应放在哪里。
5. **开发约束**：不做过度抽象、不引入无关依赖、不改无关模块。
6. **回归策略**：功能、修复、规则调整分别需要什么验证。
7. **文档更新规则**：哪些代码变更必须同步刷新知识库。
8. **日志规则**：哪些任务必须写入学习日志或技能运行日志。
9. **风险动作约束**：高风险操作需要人工确认。
10. **知识优先策略**：非 trivial 任务先查知识库，再进入实现。

## 5.2 治理层的原则定位

治理层不是为了“限制 AI 产出”，而是为了：

- 避免 AI 因为上下文不足随意发明结构。
- 把工程习惯前置成明确规则。
- 让后续所有 agent 和 skill 的行为可预测。

## 5.3 治理层与知识层的关系

治理层给出的是**边界和动作规则**；
知识层沉淀的是**工程事实和结构理解**。

治理层不负责解释所有细节，但它规定 AI 在缺少细节时应该如何行动。

## 6. 认知层蓝图

认知层的任务是：让 AI 先学工程，再碰代码。

认知层的两个核心资产是：

1. `PaperNormAI-knowledge-builder.agent.md`
2. `docs/knowledge/PaperNormAI-knowledge/`

## 6.1 `PaperNormAI-knowledge-builder.agent.md` 的定位

这是一个**知识学习智能体**，不是开发智能体。

它的使命是：

- 阅读工程结构
- 理解模块职责
- 理解架构分层
- 串联业务流程
- 识别跨层调用关系
- 把学习结果回写为公共知识

### 6.1.1 它默认不应该做的事

除非明确授权，否则它不应：

- 主动改业务代码
- 主动重构项目结构
- 直接修复缺陷
- 绕过知识库直接输出实现方案

### 6.1.2 它默认应该做的事

它应优先完成：

1. 扫描工程目录结构。
2. 识别模块边界与命名规律。
3. 理解核心业务流程。
4. 识别数据库模型与领域模型关系。
5. 理解 API 与前后端调用链路。
6. 识别知识缺口、结构冲突和未记录的架构事实。
7. 更新知识库与学习日志。

## 6.2 认知层的工作模式

推荐将知识学习拆成四种模式：

1. **总览学习**：先建立系统级认知。
2. **专题学习**：如数据库、规则引擎、API、前端状态流。
3. **流程学习**：按业务功能串调用链。
4. **增量学习**：针对本次变更区域补充知识。

### 6.2.1 总览学习

目标：建立对整个仓库的骨架认知。

输出主要更新：

- `100-system-overview.md`
- `700-capability-map.md`
- `000-doc-map.md`

### 6.2.2 专题学习

目标：把复杂子系统拆开学习。

输出主要更新：

- `200-database-models.md`
- `300-backend-kernel-services.md`
- `400-api-architecture.md`
- `500-frontend-architecture.md`
- `600-domain-models.md`

### 6.2.3 流程学习

目标：从功能入口到结果输出，串起跨层调用链路。

输出主要更新：

- `800-cross-layer-call-chains.md`
- `700-capability-map.md`

### 6.2.4 增量学习

目标：在已有知识基础上，补充新增模块或新变更点。

输出主要更新：

- 对受影响知识文档的局部刷新
- `900-learning-log.md`
- `910-skill-run-log.md`

## 7. 知识库蓝图

知识库的目标不是复制代码，而是沉淀 **代码背后的结构认知**。

## 7.1 知识库核心文档职责

### `000-doc-map.md`

作用：知识总入口。

至少应回答：

- 哪个文档讲什么。
- 哪个模块对应哪个知识文档。
- 哪个功能流程对应哪些文档。
- 哪个 agent / skill 应优先读取哪些文档。

### `010-knowledge-governance.md`

作用：规定知识库如何维护。

至少应回答：

- 哪些情况需要刷新知识库。
- 哪些文档是高优先级知识源。
- 如何处理知识与代码不一致。
- 如何记录“已确认事实”和“待确认假设”。

### `100-system-overview.md`

作用：PaperNormAI 的系统总览。

主要内容：

- 产品目标
- MVP 范围
- 仓库结构
- 核心子系统
- 高层业务流程

### `200-database-models.md`

作用：数据库模型公共认知。

主要内容：

- 关键表与用途
- 关系结构
- 业务含义
- 哪些模型服务于哪些功能链路

### `300-backend-kernel-services.md`

作用：后台核心服务公共认知。

主要内容：

- 用例服务
- 规则引擎
- 文档解析与修正服务
- AI 服务编排
- 作业系统

### `400-api-architecture.md`

作用：接口架构公共认知。

主要内容：

- API 资源分类
- 任务型接口与查询型接口
- 请求/响应模型
- 前后端交互模式

### `500-frontend-architecture.md`

作用：前端架构公共认知。

主要内容：

- 页面与 feature 划分
- 状态流
- API 调用方式
- UI 与业务状态的关系

### `600-domain-models.md`

作用：领域模型公共认知。

主要内容：

- document/template/validation/correction 等核心领域模型
- 领域模型之间的关系
- 领域对象与数据库模型的区别

### `700-capability-map.md`

作用：功能能力地图。

主要内容：

- 产品能力分类
- 每项能力落在哪些模块
- 当前能力状态（已有、规划中、未开始）
- 功能到模块的映射关系

### `800-cross-layer-call-chains.md`

作用：跨层调用链路地图。

主要内容：

- 功能入口
- API 路径
- 应用服务
- 领域服务
- 基础设施适配器
- 关键数据变换节点

### `900-learning-log.md`

作用：记录知识学习行为。

主要内容：

- 本次学习目标
- 阅读范围
- 新确认的事实
- 发现的空白或冲突
- 更新了哪些知识文档

### `910-skill-run-log.md`

作用：记录技能运行行为。

主要内容：

- 哪个 skill 被调用
- 解决了什么问题
- 读了哪些知识源
- 产出了哪些变更或判断
- 是否需要补知识库

## 7.2 知识库不是代码镜像

知识库不应做以下事情：

1. 逐文件复制代码内容。
2. 变成注释型代码备份。
3. 记录完全可以通过 grep 得到的低价值细节。

知识库应该保留的，是：

- 架构上的事实
- 分层关系
- 调用链理解
- 模块职责
- 功能与代码的映射
- 约束与例外
- 知识缺口与演进轨迹

## 7.3 知识文档的标准结构

建议所有知识文档统一采用以下结构：

```text
1. 文档目的
2. 覆盖范围
3. 核心事实
4. 关键模块 / 关键对象 / 关键流程
5. 与其他文档的关联
6. 当前已知边界
7. 待确认问题
8. 更新时间与依据
```

这样做的目的，是让 AI 在不同文档间切换时拥有稳定阅读预期。

## 8. 执行层蓝图

执行层负责在知识充分的前提下，把需求转换为实现动作。

它包含两个层次：

1. Agent：承担长期角色。
2. Skill：承担具体动作模板。

## 8.1 `PaperNormAI-feature-development.agent.md` 的定位

这是功能开发智能体，不是单纯代码生成器。

它的工作前提是：

- 先读取治理规则。
- 先读取相关知识文档。
- 先明确功能边界、影响范围和回归面。
- 再进入实现或实现建议。

### 8.1.1 它的主要职责

1. 根据需求定位涉及的模块、流程和知识入口。
2. 基于知识库判断当前实现应落在哪个分层。
3. 在不破坏既有边界的前提下进行功能实现。
4. 在完成后标记哪些知识文档需要刷新。
5. 把执行行为写入日志。

### 8.1.2 它不应做的事

1. 在知识缺失时直接发明新结构。
2. 跳过治理层的约束。
3. 忽略知识库与现有架构约束。
4. 以“能跑”为由破坏分层边界。

## 8.2 Skill 模块体系

你提出的四个 skill 非常合理，建议保留为固定基础技能：

1. `PaperNormAI-issue-evaluator`
2. `PaperNormAI-feature-readiness`
3. `PaperNormAI-feature-development`
4. `PaperNormAI-fix-development`

### 8.2.1 `PaperNormAI-issue-evaluator`

作用：任务评估与入口判断。

它至少应回答：

- 这是功能问题、缺陷问题、架构问题还是知识问题。
- 涉及哪些层：前端、API、应用服务、领域、基础设施、模板系统。
- 当前知识是否足够。
- 是否需要先调用知识学习智能体补认知。
- 风险级别与潜在回归面是什么。

### 8.2.2 `PaperNormAI-feature-readiness`

作用：开发前就绪性检查。

它至少应回答：

- 需求边界是否清楚。
- 相关模块是否已经有知识沉淀。
- 验收标准是否明确。
- 依赖条件是否满足。
- 需要刷新哪些知识文档。
- 是否已具备进入编码阶段的条件。

### 8.2.3 `PaperNormAI-feature-development`

作用：功能开发执行。

它至少应包含：

- 读取知识文档的前置步骤
- 落点模块判断
- 分层约束检查
- 变更范围控制
- 回归验证要求
- 产出后的知识更新要求

### 8.2.4 `PaperNormAI-fix-development`

作用：缺陷修复执行。

它至少应包含：

- 问题复现
- 根因定位
- 最小修复
- 回归保护
- 缺陷知识沉淀
- 是否需要更新调用链或模块认知

## 8.3 为什么 skill 要和 agent 分离

因为两者承担的职责不同：

- Agent 是稳定角色：持续承担“学习者”或“开发者”的身份。
- Skill 是动作模板：把特定任务流程固化下来。

这样可以避免所有行为都混在一个大 prompt 里，导致 AI 既学知识、又写代码、又做评估，最后失去动作边界。

## 9. 导航审计层蓝图

导航审计层负责回答两个问题：

1. 知识入口在哪里？
2. 这次动作发生了什么？

它由两类资产组成：

- 映射文档：`000-doc-map.md`
- 日志文档：`900-learning-log.md`、`910-skill-run-log.md`、`logs/*`

## 9.1 `000-doc-map.md` 的关键地位

这份文档不是普通目录索引，而是 PaperNormAI AI 工程系统的导航总线。

它至少要建立 5 种映射关系：

1. **文档 -> 主题**
2. **模块 -> 知识文档**
3. **功能 -> 模块**
4. **流程 -> 调用链文档**
5. **任务类型 -> 推荐读取顺序**

### 9.1.1 它应该支持的典型查找场景

例如：

- 我要理解“模板选择到规则加载”的流程，该看哪些文档。
- 我要改某个 API，应该先读什么。
- 我要修某个前端问题，涉及哪些状态流。
- 我要看某个模块的跨层调用链，该从哪里进。

如果 `000-doc-map.md` 不能支撑这些问题，它就不是有效的导航入口。

## 9.2 日志体系建议

推荐采用“双层日志”结构：

### 第一层：总日志文档

- `900-learning-log.md`
- `910-skill-run-log.md`

用于沉淀可长期回顾的高层摘要。

### 第二层：明细日志目录

- `logs/knowledge-builder/`
- `logs/feature-development/`
- `logs/fix-development/`
- `logs/issue-evaluator/`

用于记录单次动作的详细上下文。

## 9.3 单次日志的建议字段

每次学习或技能运行，至少记录：

```text
- 时间
- 执行角色（agent / skill）
- 触发原因
- 关联需求 / 问题
- 读取的文档
- 检查的模块
- 新确认的事实
- 发现的风险或冲突
- 更新的文档
- 后续建议动作
```

## 9.4 为什么日志必须进入知识库体系

因为如果日志只存在聊天记录里，它就不可复用、不可追踪、不可作为团队公共认知的一部分。

日志进入知识库后，至少会带来三种价值：

1. 可回放：能看清某次判断和开发为什么这么做。
2. 可追责：知道哪些知识是何时、基于什么事实更新的。
3. 可演进：后续可以基于旧日志修正认知，而不是从零开始。

## 11. 知识刷新与漂移治理

这是整套体系能否长期有效的关键问题。

## 11.1 为什么一定会漂移

只要代码继续演进，就会出现：

- 文档过时
- 调用链变化
- 模块职责变动
- 数据模型调整
- 新能力未纳入能力地图

如果没有刷新机制，知识库很快就会失去可信度。

## 11.2 刷新触发条件

建议将以下变更列为“必须评估是否更新知识库”的触发器：

1. 新增或重构核心模块。
2. API 契约变化。
3. 数据库模型变化。
4. 领域模型变化。
5. 规则引擎行为变化。
6. 模板系统结构变化。
7. 跨层调用链变化。
8. Agent / skill 行为约束变化。

## 11.3 刷新原则

1. **代码是当前事实源**：代码描述当前真实行为。
2. **蓝图是目标约束源**：蓝图描述应遵循的方向与边界。
3. **知识库是可维护认知层**：知识库负责把事实与架构意图连接起来。

当三者冲突时，应该明确记录：

- 是代码偏离了蓝图；还是
- 知识库落后于代码；还是
- 蓝图本身需要更新。

## 11.4 文档新鲜度字段建议

每份知识文档建议显式记录：

- 最近复核时间
- 复核依据（代码范围 / 提交 / 文档来源）
- 当前可信度说明
- 待确认点

这样 AI 在读取知识时，能知道哪些内容更稳定，哪些内容只是阶段性理解。

## 12. 推荐的 agent / skill 契约观

为了让这套体系长期可控，建议你把每个 agent / skill 都当作一个“工程契约”而不是一段提示词。

## 12.1 契约至少要定义的内容

每个 agent / skill 至少要定义：

1. 它的目标。
2. 它的输入前提。
3. 它必须先读什么。
4. 它允许输出什么。
5. 它不能做什么。
6. 它完成后必须回写什么。

## 12.2 典型前置读取顺序

建议固定一个默认读取优先顺序：

1. `.github/copilot-instructions.md`
2. `000-doc-map.md`
3. 与任务相关的专题知识文档
4. 与任务相关的蓝图文档
5. 任务上下文与代码

这样可以最大限度避免 AI 直接在源码局部上下文里做出全局性错误判断。

## 14. 这套体系的最终价值

如果这套 AI 工程协作体系运行良好，PaperNormAI 将获得的不只是“AI 帮忙写代码”，而是一套可长期积累的工程外部记忆系统。

它会带来以下长期价值：

1. 新任务进入前，AI 能先找到边界和入口，而不是随机发挥。
2. 同一模块的认知不会在多次任务中重复从零建立。
3. 代码实现能更贴近既有架构与分层规则。
4. 设计决策、问题判断、修复经验、调用链理解都能被保存下来。
5. 团队未来引入新的 AI 工具或新的协作成员时，也能复用这套外部系统。

换句话说，这套体系的真正产物不是某个单点 agent 或某条 skill，而是一个 **AI 可持续学习、可持续执行、可持续沉淀的工程协作环境**。

## 15. 当前建议锁定的决策

基于你目前的目标，建议现在锁定以下决策：

1. 将 AI 协作体系视为正式工程系统，而不是零散辅助工具。
2. 优先建设治理层和认知层，再建设执行层。
3. 将知识库视为工程公共认知，不只是 AI 备忘录。
4. 将 `000-doc-map.md` 视为导航总线，而不是普通目录索引。
5. 将 agent 定义为角色，将 skill 定义为动作模板。
6. 将日志纳入知识库体系，而不是只留在会话历史中。
7. 所有非 trivial 开发任务，应遵守“先知识、后开发”的默认策略。
