## 架构变更要点（2026-05-03）

### 原理解 → 新理解

**原：**
用户上传 thesis_doc → 选择系统预置模板 → 基于模板规则检测

**新：**
```
用户上传两份文档（可同时，可分开）：
  ├── spec_doc (规范手册) ──→ docling 解析 ──→ AI 语义提取规则
  └── thesis_doc (论文)    ──→ docling 解析
                                    ↓
                          AI 语义校验 ──→ ValidationReport
                                    ↓
                          Git-diff 风格展示差异
                                    ↓
                          用户手动编辑校正
                                    ↓
                          AI-Word-Skill 合并回 .docx

Fallback：用户未上传 spec_doc → 使用系统内置规则（论文规范.md）
```

### 关键技术选型

| 组件 | 选型 |
|------|------|
| 文档解析 | docling（Python 库集成） |
| 规则提取 | AI 语义理解（抽象描述性规则） |
| 规则形态 | 描述性规则，非结构化参数 |
| 文档合并 | AI-Word-Skill（只改文字，保留排版） |
| 模板系统 | Fallback 机制（用户无 spec_doc 时使用） |

### MVP 分批

| 阶段 | 覆盖内容 |
|------|---------|
| Phase 1 | 字体、字号、段落、页边距等基础文本格式 |
| Phase 2 | 公式、表格、插图格式检测与修正 |

## 关键业务理解

1. **用户上传流程** — 规范手册和论文可同时上传，分开处理
2. **规则存储** — 随用户 Session 存在，持久化支持后续复用
3. **输出结果** — Git-diff 风格展示检测报告，用户可手动编辑后确认保存
4. **公式/表格/插图** — Phase 2 再处理，Phase 1 只做基础文本格式
